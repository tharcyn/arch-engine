# Policy Patch Artifact Lifecycle

The Policy Patch Artifact is the fundamental unit of mutation in Arch-Engine. This document explains how patch artifacts are constructed, sealed, exported, verified, and safely executed across environments without compromising deterministic execution boundaries.

## What A Policy Patch Artifact Is

A policy patch artifact represents a deterministic mutation intent. It is strictly derived from an evaluation snapshot at a specific point in time. 

* The artifact is **execution-portable**, meaning it can be generated on a developer laptop and executed safely by a remote CI pipeline.
* The artifact is **replay-safe only inside matching context**, meaning its execution will actively refuse to operate if the target environment identity diverges from the intended destination.

A policy patch artifact is **not a diff**. It is **not a script**. It is a structured mutation contract carrying semantic architectural changes.

## Artifact Construction Pipeline

When an evaluation determines that a mutation is required to resolve invariant drift, the engine constructs the patch artifact. The foundation of this construction is the `evaluationContextFingerprint`.

The fingerprint is deterministically derived from:
1. **topology_dataset_identity**: To bind the mutation to the exact structural shape of the codebase it evaluated.
2. **policy pack signatures**: To bind the mutation to the exact rule definitions that demanded it.
3. **violation fingerprint surface**: To bind the mutation to the exact infractions it aims to resolve.

Selecting these inputs guarantees that if the architecture, the rules, or the violations change, the underlying fingerprint inherently changes.

## Policy File Fingerprint Binding

The artifact explicitly records a `policyFileFingerprint`. 

This binding exists to prevent stale write corruption and detect concurrent modifications. When the payload is executed, if the target repository's current policy file hash no longer matches the `policyFileFingerprint` generated during offline evaluation, execution can gracefully refuse the mutation to preserve patch correctness.

## Export Artifact Integrity Hash

To make the artifact portable, it is sealed with an `exportArtifactIntegrityHash`.

The hash calculation relies on a strict canonicalization strategy:
* **Sorted keys**: Ensures structural JSON layout does not alter the hash.
* **Semantic mutation surface only**: The hash ignores purely presentational elements (like UI strings or formatting).
* **Environment excluded**: The hash excludes local absolute paths or temporary directory names.
* **Timestamps excluded**: The hash explicitly drops chronological execution time.

This strict canonicalization ensures that the exact same violation detected in a local Mac terminal yields the mathematically identical `exportArtifactIntegrityHash` as a violation detected inside a Linux CI runner, enabling cross-agent reproducibility and pipeline safety.

## Producer Identity Binding

Every export artifact binds an `exportArtifactProducerIdentity` (e.g., `arch-engine@1.0.0`).

This ensures absolute traceability. It acts as an upfront schema compatibility validation gate, granting adapters the ability to reject payloads generated by future engine versions they do not yet understand, maintaining engine provenance clarity.

## Schema Version Binding

The payload enforces structural compatibility using an `exportArtifactSchemaVersion`. 

This provides a hard automation safety guard. Adapters parse this version string immediately upon ingestion, ensuring the payload interpretation correctness remains structurally sound across major engine evolutions.

## Repository Identity Hint Binding

During artifact construction, the engine binds the logical destination using `repositoryHint` and specifies how it derived that identity via `repositoryHintSource`.

* **env-derived**: Sourced directly from highly trusted CI variables (e.g., `GITHUB_REPOSITORY`). Treated as a strong identity.
* **package.json**: Sourced via heuristic extraction from local package manifests. Treated as an advisory identity.

Execution adapters use the source trust level to alter enforcement behavior, commonly relaxing enforcement to advisory-only during local `--dry-run` executions if the source was heuristic.

## Repository Identity Canonicalization

Before hashing or verification, the adapter subjects the hint to normalization. 

Inputs like `org/repo`, `https://github.com/org/repo`, and `git@github.com:org/repo.git` are all deterministically canonicalized down to the logical `org/repo` format. Canonicalization is required before hashing and verification to ensure stylistic variations in local git remotes do not generate artificial identity mismatches.

## Execution Plan Construction

When a provider adapter ingests the export artifact, it transforms the neutral payload into an execution plan.

The most critical derivation is the **branch naming invariant**:
`arch-engine/policy-update/<targetProfile>/<shortIntegrityHash>`

The integrity suffix exists to completely eradicate namespace collisions and parallel CI race conditions. Because the suffix is mathematically sliced from the `exportArtifactIntegrityHash`, concurrent executions generating the exact same patch will naturally converge on the exact same branch name safely.

## Execution Boundary Verification

Just before mutative network calls are allowed, the execution plan subjects itself to runtime enforcement. The adapter pulls the physical environment truth (e.g., `GITHUB_REPOSITORY` or `CI_PROJECT_PATH`) and compares it against the canonicalized payload hint.

If the payload intends to execute on `org/alpha` but the CI runner is physically executing within `org/beta`, the execution yields a strict refusal semantic, preventing catastrophic cross-repository pollution.

## Duplicate PR Suppression Model

Mutative execution follows a strict duplicate PR suppression model.

1. **Existing Branch Detection**: If the deterministic branch already exists, it is force-updated cleanly.
2. **Existing PR Detection**: The adapter queries the provider to verify if a Pull Request is already open against the target branch.
3. **Reuse Behavior**: If an active PR exists, duplicate creation is suppressed.
4. **adapterOutcome Mapping**: The pipeline cleanly maps the event to `adapterOutcome: 'pr-reused'` instead of `'pr-created'`.

## Execution Result Telemetry Contract

The entire artifact lifecycle culminates in the `AdapterExecutionResultBase` telemetry payload.

This contract provides standard, highly observable fields to the caller, abstracting away complex log scraping. The cornerstone of this contract is the `adapterOutcome` lifecycle classification.

The normalized vocabulary (`dry-run`, `refused`, `pr-created`, `pr-reused`) exists because downstream automation orchestrators require deterministic, enum-bound signals to execute conditional logic safely (e.g., sending Slack alerts on creation, but silencing them on reuse).
