# Deterministic Execution Contract — @arch-engine/core

## 1. Purpose of the Determinism Contract

The primary purpose of `@arch-engine/core` in a distributed, multi-registry environment is to evaluate architecture policies deterministically. To guarantee this, the core engine enforces a strict **semantic execution identity**.

Deterministic closure identity exists to ensure that when a specific graph of policies evaluates against a specific topology, the mathematical structure of that execution is immune to the transport conditions under which it ran.

This strict determinism contract enables:

- **Portable verification:** Execution traces computed off-site or in isolation natively prove matching behavior against centralized deployment zones.
- **Federation safety:** Resolving policy sets across disparate organizational network boundaries guarantees outcome stability.
- **Snapshot replay certification:** The semantic snapshot generated by one evaluation pipeline handles independent re-validation seamlessly.
- **CI drift detection:** Because output traces are mathematically stable, arbitrary environment noise will never break pipeline evaluations without an underlying semantic policy change.

## 2. Execution Identity Boundary Definition

> **Execution identity** = `closureGraphHash` + `authorityTier` participation + `mergeAuthority` resolution + `originPolicyChain` lineage.

Semantic identity participates directly in the fingerprint of graph outcomes. It encompasses the raw directives, rule interactions, security-enforced authorities, and final compositional inheritance graphs that form the realized enforcement policy state.

**Explicitly excluded** from execution identity:
- registry routing path
- mirror fallback activation
- filesystem location
- execution host
- signature envelope transport metadata
- timestamps
- environment variables

### Deterministic Execution Envelope

The deterministic execution envelope is the minimal input set permitted to influence semantic execution identity.

**Participating inputs:**
- policy graph structure
- overlay merge algebra participation
- authority-tier eligibility evaluation
- originPolicyChain lineage resolution
- executed rule activation topology

These inputs collectively determine `closureGraphHash` identity.

**Excluded inputs:**
- registry routing path
- mirror fallback activation
- filesystem location
- execution host characteristics
- signature envelope transport metadata
- timestamp values
- environment variables
- registry resolution order

Changes to excluded inputs MUST NOT modify semantic execution identity.

> **Invariant:** `closureGraphHash` represents the semantic fingerprint of the deterministic execution envelope and MUST remain stable when only non-participating inputs change.

`closureGraphHash` stability confirms execution identity preservation across transport environments.

#### Federation Safety Alignment

The deterministic execution envelope guarantees safe registry mirror fallback, cross-repo federation routing, snapshot replay portability, and adapter discovery extension behavior. Because `transport != semantics`, external organizational systems routing policies toward the evaluation core can safely optimize fail-overs without injecting structural taint into the resulting evaluation traces. 

#### Adapter Safety Alignment

Capability adapters operate outside the deterministic execution envelope unless explicitly modifying participating semantic inputs. Adapter discovery MAY expand topology visibility, but adapter discovery MUST NOT alter semantic execution identity.

#### Snapshot Certification Alignment

Snapshot replay certification verifies equivalence of the deterministic execution envelope, not equivalence of the transport environment. 

> **Invariant:** Snapshot replay equality confirms deterministic execution envelope preservation.

## 4. Closure Graph Hash Contract

The `closureGraphHash` represents semantic execution structure only.

**What contributes to the hash:**

- Executed overlay merges
- Evaluated policy graph (nodes, edges, scopes)
- Authority-tier effects (resolved operation abilities)
- `originPolicyChain` lineage metadata
- Rule activation structure (merge instructions enacted)

**What is explicitly excluded:**

- Registry routing instructions
- Mirror fallback telemetry and activation states
- Filesystem paths and local mount trajectories
- Execution host architectural differences and OS data
- Signature transport metadata (key IDs, enveloped structures, digests)
- Environment variables governing local node logic

> **Invariant:** The `closureGraphHash` MUST remain identical across transport-equivalent executions.

### Closure Graph Hash Stability Assertion

> The `closureGraphHash` MUST remain stable across:
> - mirror fallback routing
> - registry resolution order changes
> - execution host differences
> - filesystem relocation
> - signature envelope transport metadata changes

## 4. Transport Independence Contract

The transport layer MUST NOT affect semantic identity. How an overlay arrives at the core composition engine remains distinct from what that policy states.

Conditions that do not factor into execution identity:

- Registry resolution order
- Mirror fallback routing sequences
- Network availability state during fetch evaluations

> **Invariant:** transport != semantics.

## 5. Signature Envelope Contract

Cryptographic signature metadata participates heavily in authority evaluation. Envelopes declare a signature's shape, its underlying algorithm binding, and evaluate against trust pools. However, this metadata does NOT participate in semantic closure identity.

Authenticity influences the engine's permission to execute overlays. If an overlay passes authorization, its functional directives are admitted into the semantic processing graph. Evaluation dictates _permission to execute_, NOT the _execution identity_ itself.

> **Invariant:** Signature envelope transport metadata MUST NOT modify `closureGraphHash`.

## 6. Authority-Tier Lattice Contract

Authority tiers probabilistically constrain what an overlay can accomplish during composition.

- **Authority-tier participation is part of semantic identity.** (Executing as a `TRUSTED_POLICY_PACK` vs. a `SIGNED_EXTERNAL_PACK` changes execution boundaries).
- **Authority-tier transport origin is NOT part of identity.** (Where the engine fetched the identity definition from is ignored).

The lattice formally enforces:

- **Authority ceiling enforcement:** Constraining base allowances via registry trust levels.
- **Per-seam authority grants:** Resolving targeted elevation thresholds specifically assigned to defined functional boundaries.
- **Overlay execution eligibility:** Rejecting untrusted mutations prior to merging.

## 7. Overlay Merge Algebra Contract

The process mapping how multiple policy matrices resolve collisions into a singular graph is constrained by precision matrix resolution, whereby `mergeAuthority` resolution is deterministic.

The algebra defines strict paths for:

- **`resolved-severity`**: Escalation bounding when traversing disparate threat interpretations.
- **`additive` injection**: Applying distinct new definitions independently.
- **`suppression` behavior**: Silencing node interactions via localized domain scopes.

> **Invariant:** Overlay merge algebra MUST be replay-stable.

## 8. Registry Provenance Contract

The `originPolicyChain` participates functionally in semantic identity. It proves logical authorship trace lineages for all active rules generated through composition depth.

However, registry routing metadata does NOT participate structurally.

> **Logical provenance != transport routing.**

The fact that an overlay authored by `registry-partner/security-overlay` acted on rules is crucial semantic provenance. Evaluating whether that data was fetched from `registry-partner-mirror` via a secondary transport proxy is an operational execution detail lacking meaning.

> **Invariant:** `originPolicyChain` MUST remain stable across replay-equivalent executions.

## 9. Snapshot Replay Certification Contract

Snapshot replay equality implies mathematical semantic execution identity equality.

> **`closureGraphHash(original) == closureGraphHash(replay)` means execution identity preserved.**

If two independently generated execution artifacts share identical structural hashing footprints, the compositional state space between both environments operated exactly identically.

> **Invariant:** Snapshot replay portability is guaranteed across transport environments.

### Snapshot Schema Compatibility Window

Snapshot portability is guaranteed within a schema-compatible execution window. 

*   **Major schema version changes invalidate replay compatibility.**
*   **Minor schema revisions preserve replay compatibility.**
*   **Snapshots remain identity-valid only within a compatible schema envelope.**

If the structural AST schema defining the engine's serialization objects advances significantly, deterministic evaluation is walled off to prevent data corruption. 

> **Snapshot replay equality requires schema compatibility alignment.**

## 11. Failure Conditions

**What a `closureGraphHash` mismatch indicates:**

- A policy graph change (nodes/edges modified).
- An authority-tier change (different clearance level detected, altering execution ability).
- A merge algebra change (composition instruction alterations resulting in diverse outcomes).
- An `originPolicyChain` change (logical policy hierarchy mutation).

**What a mismatch does NOT indicate:**

- Mirror fallback activation states.
- Registry resolution mechanism alterations.
- Execution host architectural upgrades.

If hashes drift, the deterministic semantics of the environment have functionally mutated.
