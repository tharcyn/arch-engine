# Writing a Provider Adapter

This guide explains how to implement a provider adapter that safely transforms the provider-neutral `FederationEvaluationPolicyPullRequestPayload` into provider-specific mutation operations (like GitHub Pull Requests or GitLab Merge Requests) while rigorously preserving all Arch-Engine deterministic guarantees.

## What A Provider Adapter Is

Provider adapters execute mutation intent. The core Arch-Engine system remains entirely provider-neutral; it has no concept of APIs, tokens, or Pull Requests.

An adapter's responsibilities are strict:

- Enforce runtime repository identity safety.
- Enforce schema compatibility.
- Normalize execution telemetry.
- Translate intent into provider API calls securely.

## Provider Adapter Architecture Pattern

Every provider adapter must implement a canonical pipeline:

`payload`
→ `execution plan`
→ `runtime repository verification`
→ `branch construction`
→ `provider mutation`
→ `execution telemetry surface`

Adapters must ultimately map their execution results back to the `AdapterExecutionResultBase` contract and return a strictly normalized `AdapterExecutionOutcome`.

## Required Adapter Components

An adapter implementation typically splits into several distinct functional areas:

- **`parseRepositoryHint()`**: Deterministic parsing of the payload's logical repository string into provider-specific structure (e.g., extracting owner/repo). Must NOT call network APIs.
- **`buildExecutionPlan()`**: Deterministic translation of the payload into the provider's execution vocabulary. Must calculate target branches and validate hashes completely offline.
- **`verifyRuntimeRepositoryContext()`**: Environment-specific check ensuring the physical runner context (e.g., checking `CI_PROJECT_PATH`) matches the parsed repository hint. Must remain offline and deterministic.
- **`findExistingPullRequestOrMergeRequest()`**: Provider-specific network call to query existing upstream state.
- **`executePlan()`**: Provider-specific mutative phase.

_Rule of thumb:_ Only execution and PR detection are allowed to call provider APIs. Everything else must remain purely deterministic.

## Dry-Run Contract

The `dry-run` execution mode is the **default mode** for all adapters.

- `dry-run` must _never_ mutate remote state.
- `dry-run` must still perform repository verification.
- `dry-run` must return structured execution telemetry mapping to `adapterOutcome: "dry-run"`.

## Repository Identity Verification Rules

Adapters must handle the payload's `repositoryHintSource`:

- **env-derived identity**: Indicates strong identity context generated by a highly trusted upstream environment logic.
- **package.json fallback identity**: Indicates advisory heuristic context.

### Refusal Semantics

- **Strong mismatch**: If the runtime environment contradicts the hint, the adapter must abort execution.
- **Weak mismatch advisory**: If running in `dry-run` and the source was advisory, the adapter may emit a warning without terminating.
- **Execute-mode refusal behavior**: When explicitly running with `--execute`, any mismatch must yield an immediate hard block and an `adapterOutcome: "refused"` result.

See [Repository Context Verification](../security/repository-context-verification.md) and [Refusal Reasons](../contracts/refusal-reasons.md).

## Execution Plan Determinism Requirements

Execution plans must remain perfectly verifiable:

- Execution plans must be entirely serializable to JSON.
- Execution plans must avoid filesystem probing.
- Execution plans must avoid git CLI inspection.
- Execution plans must avoid attaching chronological timestamps.
- Execution plans must avoid randomness.

## Branch Naming Contract

Adapters must construct branch names utilizing the exact invariant format:
`arch-engine/policy-update/<targetProfile>/<shortIntegrityHash>`

Adapters MUST utilize `buildShortIntegritySuffix` imported from `@arch-engine/adapter-shared`. Custom hash slicing or alternative branch structures are forbidden.

See [Branch Naming Contract](../contracts/branch-naming.md).

## Duplicate Pull Request Suppression Contract

Adapters are responsible for detecting and handling pre-existing mutations to prevent queue spam.

1. **Existing branch detection**: Force-update the branch if it already exists.
2. **Existing PR detection**: Query the provider to see if an active review is already open.
3. **Reuse semantics**: Do not open a new Request if one is already open.
4. **adapterOutcome mapping**:
   - New PR → `adapterOutcome: "pr-created"`
   - Updated existing PR → `adapterOutcome: "pr-reused"`

See [Adapter Branch Reuse Policy](../contracts/adapter-branch-reuse-policy.md).

## Execution Result Contract

All returned results must extend `AdapterExecutionResultBase` from the shared layer.

You may add provider-specific extensions (e.g., `commitSha`, `pullRequestNumber`, `pullRequestUrl` for GitHub, or `mergeRequestIid` for GitLab), but the base vocabulary must remain untouched to guarantee cross-provider stability.

See [Execution Telemetry](../contracts/execution-telemetry.md).

## adapterOutcome Lifecycle Classification

The `AdapterExecutionOutcome` is restricted to exactly four values:

- `dry-run`
- `refused`
- `pr-created`
- `pr-reused`

Adapters **must not** invent provider-specific variants. Automation orchestrators expect these specific keys to drive workflow state transitions.

See [Adapter Execution Outcome Contract](../contracts/adapter-execution-outcome.md).

## Schema Version Enforcement Rules

Adapters must validate the `pullRequestPayloadSchemaVersion` present in the payload. If the adapter receives a schema version that it does not support, it must halt execution, yield `adapterOutcome: "refused"`, and emit the canonical `SCHEMA_MISMATCH` refusal reason (see [Refusal Reasons](../contracts/refusal-reasons.md)).

## Future Adapter Compatibility Expectations

The GitHub adapter (`@arch-engine/adapter-github`) serves as the strict reference implementation for all future adapters (such as GitLab or Bitbucket).

All future adapters are expected to deeply reuse the `packages/adapters/shared` layer to maintain these cross-provider lifecycle consistency guarantees.
