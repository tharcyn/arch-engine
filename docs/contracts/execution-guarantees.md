# Execution Guarantees

This document defines the formal execution guarantees provided by Arch-Engine across evaluation, artifact construction, export sealing, execution planning, provider adapter execution, and execution telemetry. 

## What This Document Defines

Execution guarantees describe which properties remain stable across environments, which properties remain deterministic across CI runs, and define safe automation integration boundaries for platform engineers and CI tooling.

These execution guarantees apply **only** to mutation intent generation and adapter execution orchestration. Execution guarantees do NOT apply to provider runtime variability (e.g., API timeouts or GitHub/GitLab internal processing races).

## Deterministic Evaluation Guarantees

Arch-Engine guarantees that the evaluation stage produces perfectly identical suggestion surfaces for identical inputs. 
- **Topology dataset identity binding**: Ties the result perfectly to the specific graph extraction state.
- **Policy pack compatibility binding**: Locks the evaluation to exact policy schema versions.
- **Violation fingerprint stability**: Creates unique structural signatures for every detected drift.
- **`evaluationContextFingerprint` reproducibility**: Guarantees offline verification predictability.

## Policy Patch Artifact Guarantees

Arch-Engine guarantees that the patch artifact encodes mutation intent deterministically:
- **`policyFileFingerprint`** guards against concurrent modification, detecting if the ruleset drifted since the artifact was authored.
- **`executionContextFingerprint`** binds the specific evaluation snapshot to the payload.

Patch artifacts are replay-safe **only** when these fingerprints strictly match the active pipeline environment.

## Export Artifact Integrity Guarantees

Arch-Engine guarantees that identical semantic mutation intent produces identical integrity hashes across varying environments.
- **`exportArtifactIntegrityHash`** cryptographically binds the semantic mutation intent.
- **Canonical hashing** explicitly excludes timestamps, presentation formatting, and environmental context (such as local file paths), ensuring transport integrity across disparate CI runners.

## Schema Version Compatibility Guarantees

Arch-Engine guarantees backward and forward compatibility enforcement:
- **`exportArtifactSchemaVersion`** enforcement at the adapter boundary ensures structural safety.
- **`pullRequestPayloadSchemaVersion`** enforcement ensures payload parsing remains strictly deterministic.

Unsupported schema versions must immediately produce structured refusal results, preventing corrupt state execution.

## Repository Identity Safety Guarantees

Arch-Engine guarantees absolute protection against cross-repository payload pollution.
- **`repositoryHint` canonicalization**: Strips transport protocols and host prefixes down to a stable identifier.
- **`repositoryHintSource` provenance classification**: Understands where the identity claim originated.

**Enforcement rules:**
- `env`-derived identity (e.g., `GITHUB_REPOSITORY`) is treated as authoritative.
- `package.json`-derived identity is treated as advisory.
- Strong mismatch (authoritative vs runtime) → Hard execution refusal.
- Weak mismatch (advisory vs runtime) → Safe dry-run advisory.
- Weak mismatch + `--execute` mode → Hard execution refusal.

## Execution Plan Determinism Guarantees

Arch-Engine guarantees that execution plans represent a purely offline state translation.
- Execution plans are strictly serializable.
- Execution plans avoid filesystem probing entirely.
- Execution plans avoid git CLI inspection.
- Execution plans avoid any form of randomness.
- Execution plans avoid timestamps.

**Branch naming invariant**: All branches generated by any adapter must strictly match:
`arch-engine/policy-update/<targetProfile>/<shortIntegrityHash>`

## Duplicate Pull Request Suppression Guarantees

Arch-Engine guarantees intelligent duplicate PR suppression natively at the adapter level.
- **Existing branch reuse detection**: The branch is safely force-updated if it already exists upstream.
- **Existing PR detection**: Provider-specific querying confirms whether a PR/MR is already active.
- **Duplicate PR prevention**: Redundant requests are intentionally blocked to preserve maintainer sanity.

**`adapterOutcome` mapping:**
- If created fresh: `pr-created`
- If intercepted and updated: `pr-reused`

## Adapter Execution Boundary Guarantees

Arch-Engine enforces strict safety postures prior to network calls:
- **Dry-run default safety model**: Adapters never mutate state implicitly.
- **Explicit `--execute` requirement**: Network mutation commands require this active consent flag.
- **Runtime repository verification enforcement**: Performed mathematically prior to provider SDK initialization.
- **Schema compatibility validation**: Validated immediately upon artifact ingestion.

## Execution Telemetry Guarantees

Arch-Engine guarantees that downstream automation receives normalized structured signals, not opaque logs.
- **`AdapterExecutionResultBase`**: The JSON payload structure remains perfectly stable across any provider (GitHub, GitLab, etc.). See [Execution Telemetry](execution-telemetry.md).
- **`AdapterExecutionOutcome` lifecycle normalization**: Enforces a strictly limited response vocabulary.

**Lifecycle classifications:**
- `dry-run`: Executed safely without mutation.
- `refused`: Execution intentionally blocked due to failed safety invariant (see [Refusal Reasons](refusal-reasons.md)).
- `pr-created`: Successfully mutated upstream and opened new Request.
- `pr-reused`: Successfully updated existing upstream Request.

Provider adapters **must not** introduce alternative lifecycle classifications under any circumstances.

## Provider Mutation Non-Guarantees

Arch-Engine explicitly defines what it *cannot* guarantee once execution crosses into the network layer:
- Provider APIs introduce uncontrollable nondeterminism (rate limits, 5xx errors).
- Network failures or TCP resets may occur.
- Branch creation races may occur within the provider's internal queue.
- Merge request ID/number ordering may vary wildly.

Despite these variables, Arch-Engine adapters act as a buffer, translating this provider variability back into normalized execution telemetry.

## CI Integration Safety Guarantees

Arch-Engine guarantees that CI pipelines may safely compose `emit-policy-pr` and provider adapter execution commands natively, without risk of silent mutation drift.
- Artifact replay protection relies on unbreakable integrity hashes.
- Repository boundary enforcement prevents leaking mutations across forks.
- Schema compatibility enforcement shields against version desyncs.
- Duplicate PR suppression enables stateless, highly repetitive cron jobs.

## Adapter Author Responsibilities

Developers creating new Arch-Engine adapters must honor these guarantees:
- Respect schema compatibility enforcement identically.
- Respect repository verification rules before touching the network.
- Preserve `adapterOutcome` lifecycle normalization identically.
- Deeply reuse `@arch-engine/adapter-shared` primitives to prevent divergent behaviors.

## Summary Of Execution Trust Model

- Offline evaluation is **deterministic**.
- Export artifacts are **cryptographically bound**.
- Execution plans are **reproducible**.
- Repository identity is **verified prior to mutation**.
- Provider adapters **normalize lifecycle telemetry**.
- Provider APIs remain **outside the deterministic boundary**.
