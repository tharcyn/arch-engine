# Policy Pack Lockfile Contract

This document formalizes the guarantees provided by the `arch-engine.lock.json` file. The lockfile is the definitive artifact ensuring deterministic reproducibility for architecture governance execution across disparate environments.

## 1. Deterministic Governance Execution Identity

The lockfile acts as the absolute execution identity for a governance run. Any deviation in the policy-pack closure, version resolutions, capabilities, or dataset shapes is treated as an invalid execution environment.

## 2. Dependency Graph Closure Guarantees

The graph resolver (`resolvePolicyPackDependencyGraph`) computes the precise mathematical closure required to execute requested entry packs.
- **Fail-Closed Missing Dependencies:** The run will not proceed if any required dependency in the resolved tree is unavailable.
- **Conflict Avoidance:** Version collisions or explicit conflicts halt lockfile generation.
- **Optional Dependencies:** Handled opportunistically but recorded in the closure diagnostically if missing.

## 3. SemVer Resolution Determinism

Resolution explicitly records the exact resolved versions generated by traversing semantic ranges against the local `PolicyPackRegistry`. Once written to the lockfile, semantic ranges (`^1.2.0`) are permanently pinned to exact semantic identities (`1.2.5`).

## 4. Federation Compatibility Hash Pinning

To guarantee execution safety in multi-provider environments, the lockfile hashes and pins:
- `capabilityIntersectionHash`: The exact set of operational capabilities provided by the ingested datasets.
- `datasetCompatibilityHash`: The structural shapes supported by the environment.
- `executionModeHash`: The operational topography (`single-provider` or `multi-provider-federated`).

## 5. Registry Replay & CI Reproducibility

The `verifyPolicyPackLockfileReplay` validation engine ensures that any Continuous Integration pipeline executing `arch-engine` will execute identically to the original lockfile generator environment.
If capability intersections shift (e.g., an adapter is removed) or pack hashes mutate, the CLI returns deterministic exit codes (`2`, `3`, `4`, or `5`) halting CI and preventing drift execution.
