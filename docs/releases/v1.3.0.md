# Arch-Engine v1.3.0

**Released:** 2026-05-13
**Headline:** pnpm workspace adapter support, runtime adapter selection, JSON v2 `data.adapter`.

## What's new

v1.3.0 is the first multi-adapter Arch-Engine release. The CLI now
performs deterministic adapter selection at runtime, and a new
optional package — `@arch-engine/adapter-pnpm@0.1.0` — handles
pnpm-managed monorepos with full glob expansion, exclusion-glob
support, and the `workspace:*` family of dependency protocols.

The release is fully additive on top of v1.2.0:

- JSON v1 default output is byte-identical to v1.2.0.
- The five-command surface (`doctor`, `inspect`, `analyze`,
  `check`, `explain`) is unchanged.
- No new CLI flags or commands.
- `@arch-engine/adapter-monorepo`'s public API
  (`runMonorepoExtraction`, `classifyAuthorityDomain`,
  `monorepoAdapter`) remains byte-identical.

## Install

For an npm/yarn workspace, no change is required:

```bash
npm install --save-dev @arch-engine/cli@1.3.0 @arch-engine/adapter-monorepo@1.3.0
```

For a pnpm workspace, add the new adapter:

```bash
npm install --save-dev @arch-engine/cli@1.3.0 @arch-engine/adapter-monorepo@1.3.0 @arch-engine/adapter-pnpm@0.1.0
```

The pnpm adapter is an optional peer dependency. The CLI works
without it — pnpm workspaces fall back to the monorepo adapter's
line-based pnpm parser with an `ARCH_ENGINE_ADAPTER_LOW_CONFIDENCE`
advisory recommending the install.

## JSON v2 `data.adapter`

Every topology-extracting command's JSON v2 envelope now includes
an additive `data.adapter` block:

```jsonc
{
  "command": "inspect",
  "data": {
    "adapter": {
      "name": "@arch-engine/adapter-pnpm",
      "version": "0.1.0",
      "packageManager": "pnpm",
      "workspaceKind": "pnpm-workspace",
      "confidence": "HIGH",
      "reasons": [
        "pnpm-workspace.yaml present",
        "pnpm-lock.yaml present",
        "package.json#packageManager starts with pnpm@"
      ],
      "warnings": [],
      "alsoDetected": [],
      "metadata": {
        "pnpm": {
          "workspaceFile": "pnpm-workspace.yaml",
          "packageManagerVersion": "pnpm@9.0.0",
          "lockfilePresent": true,
          "catalogsDetected": false,
          "excludedGlobs": [],
          "matchedGlobs": ["apps/api", "apps/web", "packages/shared"]
        },
        "edges": {
          "e_<8hex>": { "kind": "dev", "protocol": "workspace" }
        }
      }
    },
    "topology": { /* canonical (nodes, edges) — unchanged */ }
  }
}
```

The block is present on `doctor`, `inspect`, `analyze`, `check`,
and `explain` (matched, unmatched, and `policy` modes). It is
intentionally absent on `explain regression`, which reads
`.arch-engine/stability-score.json` without invoking adapter
selection.

JSON v1 default output is unaffected — no `adapter` key appears.

## Six new error codes

The `ARCH_ENGINE_*` vocabulary grows from 16 to 22:

| Code | Severity | Exit | Surfaced when |
| --- | --- | --- | --- |
| `ARCH_ENGINE_ADAPTER_CONFLICT` | ERROR | 3 | Two or more adapters report HIGH confidence on the same repo. |
| `ARCH_ENGINE_ADAPTER_LOW_CONFIDENCE` | WARNING | 0 | Best detection was LOW; fallback adapter selected. |
| `ARCH_ENGINE_WORKSPACE_GLOBS_INVALID` | ERROR | 3 | `pnpm-workspace.yaml` cannot be parsed or yields no globs. |
| `ARCH_ENGINE_WORKSPACE_PACKAGE_UNNAMED` | ERROR | 3 | A workspace package's `package.json` lacks a `name` field. |
| `ARCH_ENGINE_LOCKFILE_UNSUPPORTED` | WARNING | 0 | A lockfile feature (e.g. pnpm catalogs) was detected but not interpreted by the current adapter version. |
| `ARCH_ENGINE_PNP_RESOLUTION_DEFERRED` | WARNING | 0 | Yarn PnP `.pnp.cjs` detected; v0.1.0 adapter does not execute it. (Reserved for the future yarn-pnp adapter.) |

## pnpm support — what's in / what's deferred

### In v0.1.0 of `@arch-engine/adapter-pnpm`

- `packages: [...]` array form in `pnpm-workspace.yaml`.
- Single-segment globs (`apps/*`, `packages/*`).
- Two-segment globs (`packages/*/*`).
- Exclusion globs (`!packages/private-*`).
- Always-ignored directories: `node_modules`, `.git`, `dist`,
  `build`, `.turbo`, `.next`, `.arch-engine`, `.pnpm-store`.
- `workspace:*`, `workspace:^`, `workspace:~`, `workspace:<version>`
  protocols.
- All four dependency kinds:
  `dependencies` / `devDependencies` / `peerDependencies` /
  `optionalDependencies`.
- Per-edge metadata in `data.adapter.metadata.edges.<id>`:
  `{ kind, protocol }`.
- `package.json#packageManager` hint and `pnpm-lock.yaml` presence
  detection (strengthen confidence to HIGH).

### Deferred / honest gaps

| Feature | Deferred to | Notes |
| --- | --- | --- |
| `node_modules/` / `.pnpm-store/` parsing | Out of MVP | Adapter is pure-fs read of manifests only. |
| Deep `pnpm-lock.yaml` analysis | v1.4+ | Topology comes from `package.json` edges; lockfile parsing not necessary for graph extraction. |
| `catalog:` / `catalogs:` resolution | v1.4+ | Detected and surfaced as `ARCH_ENGINE_LOCKFILE_UNSUPPORTED` INFO diagnostic; specifiers treated as opaque. |
| Recursive `**` globs in inclusion | v1.4+ | Soft-warning skip; exclusion patterns honour `**`. |
| Flow-style YAML lists / object-shape `packages:` | v1.4+ | Only block-list form supported. |
| Yarn PnP / `.pnp.cjs` execution | Separate Pass 3 (never executes repo JS) | Code-execution safety invariant. |
| `pnpm` CLI invocation | Never | Adapter never shells out. |

## `graphSurfaceHash` parity

The canonical topology hash (`data.topology.canonical.graphSurfaceHash`)
remains the source of truth for baseline drift comparison. v1.3.0
preserves the spec §12.4 invariant: **adapter identity does not
affect the hash.**

Concretely:

- Within a fixture's pnpm-glob-matched packages, the pnpm adapter
  and the monorepo adapter's pnpm-fallback path produce IDENTICAL
  canonical (nodes, edges) tuples.
- A documented asymmetry remains: the monorepo adapter includes the
  repo root as a workspace node when its `package.json` has a
  `name`; the pnpm adapter follows pnpm's documented behaviour and
  includes only `packages:`-glob-matched directories. This means
  the full-topology hash differs only by the root-inclusion delta
  — pinned by test
  (`adapter-graph-surface-hash-parity.test.ts`).
- Pnpm-only features (two-segment globs, exclusion globs,
  `workspace:*` protocols) legitimately produce different output
  shapes; both sides remain deterministic on repeat runs; no parity
  is claimed.

Baselines generated under v1.2.0 against pnpm workspaces continue
to compare cleanly against v1.3.0 outputs **for the intersection
node/edge set**. Teams that want strict bit-for-bit cross-adapter
parity on root-inclusive hashes should standardise on one adapter
or strip the root node from one side.

## Migration notes

- **Existing users (npm / yarn workspaces):** no action needed.
  Upgrade with `npm install --save-dev @arch-engine/cli@1.3.0
  @arch-engine/adapter-monorepo@1.3.0`. Behaviour is byte-identical
  to v1.2.0 except for the new `doctor` Adapter line and the
  additive JSON v2 `data.adapter` block.
- **Existing pnpm users:** also no action needed for v1.3.0 to
  work. To pick up the richer pnpm handling (`workspace:*`
  resolution, exclusion globs, two-segment glob support), install
  the new adapter:
  `npm install --save-dev @arch-engine/adapter-pnpm@0.1.0`. The CLI
  picks it up automatically.
- **CI users:** existing GitHub Actions templates work unchanged.
  Optional one-line update in `examples/github-actions/README.md`
  shows how to install `@arch-engine/adapter-pnpm` if you want the
  richer pnpm support in CI too.
- **Programmatic consumers** of `@arch-engine/adapter-monorepo`:
  no change required. `runMonorepoExtraction(cwd)` returns the
  same shape it did in v1.2.0.
