# Arch-Engine v1.4.0

**Released:** 2026-05-13
**Headline:** Yarn Berry / Plug'n'Play adapter support — safe, deterministic, never executes `.pnp.cjs`.

## What's new

v1.4.0 ships **first-class Yarn PnP workspace support** via a new
optional adapter package, [`@arch-engine/adapter-yarn-pnp@0.1.0`](../../packages/adapter-yarn-pnp/README.md).
The CLI's deterministic adapter registry now picks the right
adapter on every JavaScript workspace shape — pnpm, Yarn PnP,
Yarn classic, npm — without surfacing conflict diagnostics.

The release is fully additive on top of v1.3.1:

- JSON v1 default output is byte-identical to v1.3.1.
- The five-command surface (`doctor`, `inspect`, `analyze`,
  `check`, `explain`) is unchanged.
- No new CLI flags or commands.
- `@arch-engine/adapter-pnpm@0.1.1` and
  `@arch-engine/adapter-monorepo@1.3.1` consumers see no behavior
  change unless they also install the new yarn-pnp adapter.

Real-repo trial against 11 public OSS repos returned
`YARN_PNP_REAL_REPO_TRIAL_STRONG_SIGNAL` — **11/11 correct adapter
selections, 0 P0/P1/P2 issues, 0 path leaks, 0 repo mutations**.

## Install

For npm / yarn classic workspaces, no change is required:

```bash
npm install --save-dev \
  @arch-engine/cli@1.4.0 \
  @arch-engine/adapter-monorepo@1.3.1
```

For pnpm workspaces:

```bash
npm install --save-dev \
  @arch-engine/cli@1.4.0 \
  @arch-engine/adapter-monorepo@1.3.1 \
  @arch-engine/adapter-pnpm@0.1.1
```

For **Yarn Berry / Plug'n'Play** workspaces, install the new
adapter:

```bash
npm install --save-dev \
  @arch-engine/cli@1.4.0 \
  @arch-engine/adapter-monorepo@1.3.1 \
  @arch-engine/adapter-pnpm@0.1.1 \
  @arch-engine/adapter-yarn-pnp@0.1.0
```

All adapter packages are optional peer dependencies of
`@arch-engine/cli`. The CLI's dispatcher reaches for whichever
ones are installed.

## What changed

### New package: `@arch-engine/adapter-yarn-pnp@0.1.0`

Safe, deterministic workspace topology extractor for Yarn Berry
repos that ship `.pnp.cjs` or `.pnp.loader.mjs`. Supports:

- **Workspace shapes:** array (`"workspaces": ["packages/*"]`)
  and object (`"workspaces": { "packages": [...] }`) forms.
- **Dependency protocols:** `workspace:*`, `workspace:^`,
  `workspace:~`, `workspace:<version>`, `portal:<path>`,
  `link:<path>` — all four dependency kinds (`dependencies`,
  `devDependencies`, `peerDependencies`,
  `optionalDependencies`).
- **Corepack `packageManager` parsing:** `yarn@4.0.2+sha256.<hash>`
  → `packageManagerVersion: "4.0.2"` (integrity suffix stripped).
- **Always-emitted informational diagnostic:**
  `ARCH_ENGINE_PNP_RESOLUTION_DEFERRED` whenever a PnP file is
  present — explains that full PnP resolver parity is
  intentionally deferred and that topology comes from
  `package.json#workspaces` only.

### Registry precedence

| # | Adapter | Wins on |
| --- | --- | --- |
| 2 | `@arch-engine/adapter-pnpm` | `pnpm-workspace.yaml` present |
| 3 | `@arch-engine/adapter-yarn-pnp` | `.pnp.cjs` / `.pnp.loader.mjs` present (and no `pnpm-workspace.yaml`) |
| 4 | `@arch-engine/adapter-monorepo` | Fallback: npm / yarn-classic workspaces; single-package; or any of the above when the dedicated adapter isn't installed |

Cache-hint declines (`archengine:pnpmAdapterAvailable`,
`archengine:yarnPnpAdapterAvailable`) prevent
`ARCH_ENGINE_ADAPTER_CONFLICT` even when two adapters could both
plausibly claim a repository.

### `@arch-engine/adapter-monorepo` 1.3.0 → 1.3.1

Patch bump. Adds the yarn-pnp cache-hint decline protocol so
plain `package.json#workspaces` repos that *also* ship a
`.pnp.cjs` are handed off to the yarn-pnp adapter without
surfacing a conflict diagnostic. The check is a no-op when the
yarn-pnp adapter is not loaded — consumers who only installed
adapter-monorepo see byte-identical v1.3.0 behavior.

## Safety model

The Yarn PnP MVP **never**:

- executes `.pnp.cjs` or `.pnp.loader.mjs`
- invokes `yarn` or any package-manager binary
- runs `npm install` / `pnpm install` / `yarn install`
- opens network sockets
- reads `node_modules/`, `.yarn/cache`, `.yarnpkg/unplugged`,
  `.yarn/install-state.gz`
- mutates the target repository
- emits absolute paths in JSON v2 output

These invariants are pinned by structural tests and verified
against `yarnpkg/berry` itself (the only mainstream OSS repo
still using PnP at runtime as of 2026-05).

## JSON v2 example — `data.adapter` on a Yarn PnP repo

Running `arch-engine inspect --json --json-schema=v2` from a Yarn
Berry repository root yields a `data.adapter` block like:

```jsonc
{
  "name": "@arch-engine/adapter-yarn-pnp",
  "version": "0.1.0",
  "packageManager": "yarn",
  "workspaceKind": "yarn-pnp",
  "confidence": "HIGH",
  "reasons": [
    ".pnp.cjs present",
    ".pnp.loader.mjs present",
    ".yarnrc.yml present",
    "package.json#packageManager identifies yarn",
    "package.json#workspaces is set"
  ],
  "warnings": [],
  "alsoDetected": [],
  "metadata": {
    "yarnPnp": {
      "packageManagerVersion": "4.2.1",
      "pnpFilePresent": true,
      "pnpLoaderPresent": true,
      "yarnrcPresent": true,
      "nodeLinker": "pnp",
      "nodeLinkerSource": "yarnrc",
      "workspacesPresent": true,
      "workspacesObjectForm": false,
      "rawGlobs": ["packages/*"],
      "excludedGlobs": [],
      "matchedGlobs": ["packages/…"]
    },
    "edges": { "e_<hex8>": { "kind": "dependency", "protocol": "workspace" } }
  }
}
```

### `nodeLinkerSource` provenance

The `nodeLinkerSource` enum disambiguates how `nodeLinker` was
determined:

| Source | When | `nodeLinker` |
| --- | --- | --- |
| `"yarnrc"` | `.yarnrc.yml` explicitly declared `nodeLinker:` | the literal value |
| `"inferred_from_pnp_file"` | no explicit value, but `.pnp.cjs` / `.pnp.loader.mjs` is present | `"pnp"` (Yarn Berry's documented default) |
| `"absent"` | neither signal present | `null` |

Added in the v0.1.1 trust-polish that landed alongside this
release after the real-repo trial flagged `nodeLinker: null` on
`yarnpkg/berry` as confusing for users.

## Limitations

- **No full PnP resolver.** External `node:` requires that go
  through `.pnp.cjs` at runtime are not modelled. Workspace
  topology IS extracted accurately from `package.json#workspaces`
  and the workspace manifests.
- **Yarn PnP is rare in mainstream OSS.** Most modern Yarn Berry
  projects use `nodeLinker: node-modules` — the v1.3.1
  monorepo adapter correctly handles those as plain
  yarn-classic-style workspaces. The trial confirmed `yarnpkg/berry`
  is currently the only widely-deployed public PnP repo.
- **Unnamed-package semantic alignment deferred.** The yarn-pnp
  adapter silently drops workspace packages that lack a `name`
  field (consistent with adapter-pnpm v0.1.x behavior); future
  hardening will surface `ARCH_ENGINE_WORKSPACE_PACKAGE_UNNAMED`
  (ERROR) instead. See the real-repo trial audit §11 for detail.
- **Auto-init cache.** The CLI continues to create a small
  `.arch-engine/` cache directory inside target repos on first
  run (pre-existing v1.0.1+ behavior). v1.4.0 ships a
  `.gitignore` rule so fixture test runs no longer dirty the
  arch-engine repo itself.

## Real-repo trial summary

11 public OSS repositories probed via local-packed tarballs
(`yarnpkg/berry`, `backstage/backstage`, `babel/babel`,
`changesets/changesets`, `facebook/react`, `nrwl/nx`, `h3js/h3`,
`prisma/prisma`, `graphql/graphql-js`, `expressjs/express`,
`egoist/tsup`).

**11 / 11 correct adapter selections:**

- 1 Yarn PnP (`yarnpkg/berry`) → `@arch-engine/adapter-yarn-pnp` HIGH.
- 2 Yarn Berry `nodeLinker: node-modules` (`backstage`, `babel`) → monorepo HIGH.
- 2 Yarn classic (`react`, `changesets`) → monorepo HIGH.
- 3 pnpm workspaces (`nrwl/nx`, `h3`, `prisma`) → pnpm HIGH.
- 3 single-package fallback (`graphql-js`, `express`, `tsup`) → monorepo LOW.

Zero P0/P1/P2 issues. Full report:
[`audits/ARCH_ENGINE_ADAPTER_PASS_3_YARN_PNP_REAL_REPO_TRIAL.md`](../../audits/ARCH_ENGINE_ADAPTER_PASS_3_YARN_PNP_REAL_REPO_TRIAL.md).

## Migration

| Consumer | Action |
| --- | --- |
| v1.3.1 with `@arch-engine/cli` + `@arch-engine/adapter-monorepo` only | None. Upgrade to v1.4.0 / 1.3.1; behavior identical. |
| v1.3.1 with `@arch-engine/adapter-pnpm` | None. pnpm precedence unchanged. |
| Yarn Berry / PnP user | Install `@arch-engine/adapter-yarn-pnp@0.1.0` alongside the CLI. |
| Yarn Berry `nodeLinker: node-modules` user | None. Continues to route through `@arch-engine/adapter-monorepo` at HIGH. |
| GitHub Actions workflow templates | None — the existing workflows continue to work; install the yarn-pnp adapter in the workflow's `npm install` step if your CI repo is PnP. |
